### User Story 1 - 그룹 생성 및 기본 관리 (우선순위: P1)

인증된 사용자 김철수는 새로운 협업 프로젝트를 위해 "마케팅팀 2026"이라는 그룹을 생성합니다. 그룹 생성 시 김철수는 자동으로 그룹장이 되며, 그룹 이름과 설명을 설정할 수 있습니다. 이후 김철수는 그룹 정보를 수정하거나 필요시 그룹을 삭제할 수 있습니다.

**우선순위 이유**: 그룹 생성은 모든 그룹 관련 기능의 기반이 되는 핵심 기능입니다. 이것이 없으면 다른 어떤 그룹 기능도 사용할 수 없습니다.

**독립 테스트**: 사용자가 로그인 후 그룹을 생성하고, 그룹 이름과 설명을 수정하며, 본인이 그룹장임을 확인할 수 있습니다. 이 기능만으로도 기본적인 그룹 관리 가치를 제공합니다.

**수락 시나리오**:

1. **Given** 사용자가 로그인되어 있고, **When** 고유한 그룹 이름 "마케팅팀 2026"과 설명을 입력하여 그룹을 생성하면, **Then** 그룹이 성공적으로 생성되고 사용자는 자동으로 그룹장이 됩니다.
2. **Given** 사용자가 그룹장이고, **When** 그룹 이름을 "마케팅팀 2026 Q1"로 변경하면, **Then** 그룹 이름이 업데이트되고 모든 멤버에게 표시됩니다.
3. **Given** 사용자가 그룹장이고, **When** 그룹 삭제를 요청하면, **Then** 그룹이 보관(soft-delete) 처리되고 일반 사용자는 접근할 수 없지만 데이터는 보존됩니다.
4. **Given** 사용자가 그룹을 생성하려 하고, **When** 이미 존재하는 그룹 이름을 입력하면, **Then** 시스템은 "이미 사용 중인 그룹 이름입니다"라는 오류 메시지를 표시하고 그룹 생성을 거부합니다.

### User Story 2 - 멤버 초대 및 관리 (우선순위: P2)

그룹장 김철수는 팀원 이영희와 박민수를 "마케팅팀 2026" 그룹에 초대합니다. 이영희와 박민수는 초대를 수락하고 그룹 멤버가 됩니다. 김철수는 언제든지 그룹 멤버 목록을 확인할 수 있으며, 필요시 특정 멤버를 그룹에서 제거할 수 있습니다.

**우선순위 이유**: 그룹의 핵심 가치는 협업입니다. 멤버를 초대하고 관리할 수 있어야 그룹이 협업 공간으로 기능할 수 있습니다.

**독립 테스트**: 그룹장이 다른 사용자를 초대하고, 멤버 목록을 확인하며, 필요시 멤버를 제거할 수 있습니다. 기본 그룹 생성 기능(P1) 위에서 독립적으로 작동합니다.

**수락 시나리오**:

1. **Given** 김철수가 그룹장이고, **When** 이영희의 사용자 ID로 초대를 보내면, **Then** 이영희는 그룹 멤버로 추가되고 그룹에 접근할 수 있습니다.
2. **Given** 김철수가 그룹장이고, **When** 멤버 목록을 조회하면, **Then** 모든 멤버의 이름, 역할(그룹장/부그룹장/일반 멤버), 가입일이 표시됩니다.
3. **Given** 김철수가 그룹장이고 박민수가 일반 멤버이며, **When** 박민수를 그룹에서 제거하면, **Then** 박민수는 그룹에서 제거되고 더 이상 그룹에 접근할 수 없습니다.
4. **Given** 이영희가 일반 멤버이고, **When** 다른 사용자를 초대하려 하면, **Then** 시스템은 "권한이 없습니다"라는 오류를 표시하고 초대를 거부합니다.
5. **Given** 김철수가 그룹장이고, **When** 이미 삭제된 사용자를 초대하려 하면, **Then** 시스템은 "존재하지 않는 사용자입니다"라는 오류를 표시합니다.

### User Story 3 - 부그룹장 지정 및 세부 권한 부여 (우선순위: P3)

그룹장 김철수는 신뢰하는 멤버 이영희를 부그룹장으로 지정하고, 멤버 관리 권한과 게시판 생성 권한을 부여합니다. 이영희는 이제 새로운 멤버를 초대하고 게시판을 만들 수 있지만, 그룹 자체를 수정하거나 삭제할 수는 없습니다. 김철수는 필요시 이영희의 부그룹장 권한을 회수할 수 있습니다.

**우선순위 이유**: 권한 위임은 대규모 그룹 관리에 필수적이지만, 기본적인 그룹 운영은 그룹장만으로도 가능합니다. 이는 확장성을 위한 기능입니다.

**독립 테스트**: 그룹장이 멤버를 부그룹장으로 지정하고, 특정 권한을 부여하며, 부그룹장이 해당 권한 내에서만 작업할 수 있는지 확인할 수 있습니다.

**수락 시나리오**:

1. **Given** 김철수가 그룹장이고 이영희가 일반 멤버이며, **When** 이영희를 부그룹장으로 지정하고 "멤버 관리" 권한을 부여하면, **Then** 이영희는 부그룹장이 되고 다른 사용자를 초대하거나 제거할 수 있습니다.
2. **Given** 이영희가 "멤버 관리" 권한을 가진 부그룹장이고, **When** 새로운 사용자 최준호를 초대하면, **Then** 최준호가 그룹 멤버로 추가됩니다.
3. **Given** 이영희가 부그룹장이지만 "게시판 생성" 권한이 없고, **When** 게시판 생성을 시도하면, **Then** 시스템은 "권한이 없습니다"라는 오류를 표시합니다.
4. **Given** 이영희가 부그룹장이고, **When** 그룹 이름을 변경하려 하면, **Then** 시스템은 "그룹장만 그룹 정보를 수정할 수 있습니다"라는 오류를 표시합니다.
5. **Given** 이영희가 부그룹장이고 박민수도 부그룹장이며, **When** 이영희가 박민수를 제거하려 하면, **Then** 시스템은 "부그룹장은 다른 부그룹장을 제거할 수 없습니다"라는 오류를 표시합니다.
6. **Given** 김철수가 그룹장이고 이영희가 부그룹장이며, **When** 이영희의 부그룹장 권한을 회수하면, **Then** 이영희는 일반 멤버로 전환되고 부그룹장 권한을 잃습니다.

### User Story 4 - 그룹장 양도 (우선순위: P4)

그룹장 김철수는 프로젝트를 떠나기 전에 부그룹장 이영희에게 그룹장 권한을 양도하려 합니다. 김철수는 이영희에게 양도 요청을 보내고, 이영희는 요청을 검토한 후 수락합니다. 이영희가 수락하면 즉시 그룹장이 되고, 김철수는 일반 멤버가 됩니다. 만약 이영희가 수락 전에 김철수가 마음을 바꾸면, 김철수는 요청을 취소할 수 있습니다.

**우선순위 이유**: 그룹장 양도는 장기적인 그룹 지속성을 위해 중요하지만, 초기 그룹 운영에는 필수적이지 않습니다. 대부분의 일상적인 그룹 활동은 이 기능 없이도 가능합니다.

**독립 테스트**: 그룹장이 다른 멤버에게 양도 요청을 보내고, 대상자가 수락/거절하며, 수락 시 권한이 정상적으로 이전되는지 확인할 수 있습니다.

**수락 시나리오**:

1. **Given** 김철수가 그룹장이고 이영희가 멤버이며, **When** 김철수가 이영희에게 그룹장 양도 요청을 보내면, **Then** 이영희는 양도 요청 알림을 받고 수락 또는 거절할 수 있습니다.
2. **Given** 이영희가 양도 요청을 받았고, **When** 이영희가 요청을 수락하면, **Then** 이영희는 그룹장이 되고 김철수는 일반 멤버가 되며 모든 멤버에게 알림이 전송됩니다.
3. **Given** 이영희가 양도 요청을 받았고, **When** 이영희가 요청을 거절하면, **Then** 김철수는 그룹장으로 유지되고 양도 요청은 취소됩니다.
4. **Given** 김철수가 이영희에게 양도 요청을 보냈고 아직 대기 중이며, **When** 김철수가 요청을 취소하면, **Then** 양도 요청이 무효화되고 이영희는 더 이상 응답할 수 없습니다.
5. **Given** 김철수가 이영희에게 양도 요청을 보냈고, **When** 김철수가 그룹을 떠나려 하면, **Then** 시스템은 "대기 중인 양도 요청이 있습니다. 먼저 요청을 취소하거나 완료해주세요"라는 오류를 표시합니다.
6. **Given** 김철수가 이영희에게 양도 요청을 보냈고, **When** 김철수가 박민수에게도 양도 요청을 보내려 하면, **Then** 시스템은 "동시에 한 명에게만 양도 요청을 보낼 수 있습니다"라는 오류를 표시합니다.

### 엣지 케이스 (Edge Cases)

- **그룹장이 양도 요청 후 강제로 그룹을 떠나면?** 양도 요청은 자동으로 취소되고, 그룹에 다른 부그룹장이 있다면 가장 오래된 부그룹장이 자동으로 그룹장이 됩니다. 부그룹장이 없고 일반 멤버만 있다면, 가장 오래된 멤버가 그룹장이 됩니다.
- **양도 요청이 30일 이상 대기 상태로 남으면?** 시스템은 30일 후 자동으로 양도 요청을 취소하고 요청자에게 알림을 보냅니다.
- **삭제된 사용자가 만든 그룹은?** 그룹은 유지되며, 가장 오래된 부그룹장이 자동으로 그룹장이 됩니다. 부그룹장이 없으면 가장 오래된 멤버가 그룹장이 됩니다.
- **모든 그룹장과 부그룹장이 그룹을 떠나면?** 남은 멤버 중 가장 오래된 멤버가 자동으로 그룹장이 됩니다. 멤버가 아무도 없으면 그룹은 자동으로 보관(archived) 처리됩니다.
- **보관된 그룹 복원 시 일부 멤버가 이미 삭제된 경우?** 삭제된 사용자는 멤버 목록에서 제외되고, 나머지 활성 멤버만 복원됩니다. 그룹장이 삭제된 경우 가장 오래된 활성 멤버가 그룹장이 됩니다.
- **부그룹장이 다른 부그룹장을 제거하려 하면?** 시스템은 권한 오류를 반환하고 작업을 차단합니다. 오직 그룹장만 부그룹장을 제거할 수 있습니다.
- **동시에 여러 그룹장 권한 변경이 발생하면?** 시스템은 낙관적 잠금(optimistic locking) 메커니즘을 사용하여 마지막 변경이 우선하며, 충돌 시 사용자에게 "최신 정보로 새로고침 후 다시 시도하세요"라는 메시지를 표시합니다.
- **그룹 이름 고유성 검증이 동시 요청으로 우회되면?** 시스템은 데이터베이스 수준의 고유 제약 조건을 사용하여 동시 요청이라도 중복 그룹 이름 생성을 방지합니다.
- **부그룹장이 본인을 그룹에서 제거하려 하면?** 시스템은 이를 허용하고 해당 부그룹장을 일반 사용자로 변환한 후 그룹에서 제거합니다.
- **그룹장이 본인을 그룹에서 제거하려 하면?** 시스템은 "그룹장은 그룹을 떠나기 전에 다른 멤버에게 그룹장을 양도해야 합니다"라는 오류를 표시합니다. 본인이 유일한 멤버인 경우에만 그룹 삭제가 가능합니다.

## 요구사항 *(필수)*

### 기능 요구사항 (Functional Requirements)

#### 그룹 생성 및 기본 관리

- **FR-001**: 시스템은 인증된 사용자가 고유한 이름과 설명으로 그룹을 생성할 수 있도록 해야 합니다.
- **FR-002**: 시스템은 그룹 이름의 시스템 전체 고유성을 보장해야 합니다(대소문자 구분 없음).
- **FR-003**: 시스템은 그룹 생성자를 자동으로 그룹장으로 지정해야 합니다.
- **FR-004**: 그룹장은 그룹 이름과 설명을 수정할 수 있어야 합니다.
- **FR-005**: 그룹장은 그룹을 삭제할 수 있어야 하며, 삭제 시 soft-delete(보관) 방식을 사용해야 합니다.
- **FR-006**: 보관된 그룹의 모든 데이터는 보존되어야 하며 일반 사용자는 접근할 수 없어야 합니다.
- **FR-007**: 시스템 관리자는 보관된 그룹을 복원할 수 있어야 합니다.

#### 멤버 관리

- **FR-008**: 그룹장은 사용자 ID를 통해 다른 사용자를 그룹에 초대할 수 있어야 합니다.
- **FR-009**: 그룹장은 일반 멤버를 그룹에서 제거할 수 있어야 합니다.
- **FR-010**: 시스템은 모든 그룹 멤버 목록을 표시해야 하며, 각 멤버의 역할(그룹장/부그룹장/일반 멤버)과 가입일을 포함해야 합니다.
- **FR-011**: 시스템은 각 멤버의 그룹 내 역할을 추적해야 합니다.
- **FR-012**: 시스템은 삭제된 사용자를 초대하려는 시도를 거부하고 적절한 오류 메시지를 표시해야 합니다.
- **FR-013**: 시스템은 이미 그룹 멤버인 사용자를 다시 초대하려는 시도를 거부해야 합니다.

#### 부그룹장 및 권한 시스템

- **FR-014**: 그룹장은 일반 멤버를 부그룹장으로 지정할 수 있어야 합니다.
- **FR-015**: 그룹장은 부그룹장에게 다음 세부 권한을 개별적으로 부여할 수 있어야 합니다:
  - 멤버 관리 권한 (멤버 초대 및 제거)
  - 게시판 생성 권한
  - 콘텐츠 관리 권한
- **FR-016**: 부그룹장은 그룹 정보(이름, 설명)를 수정할 수 없어야 합니다.
- **FR-017**: 부그룹장은 그룹을 삭제할 수 없어야 합니다.
- **FR-018**: 부그룹장은 그룹장 양도를 시작할 수 없어야 합니다.
- **FR-019**: "멤버 관리" 권한이 있는 부그룹장은 일반 멤버만 제거할 수 있으며, 다른 부그룹장이나 그룹장은 제거할 수 없어야 합니다.
- **FR-020**: 그룹장은 언제든지 부그룹장의 권한을 회수하여 일반 멤버로 전환할 수 있어야 합니다.
- **FR-021**: 시스템은 부여된 권한이 없는 작업 시도를 차단하고 "권한이 없습니다"라는 오류 메시지를 표시해야 합니다.

#### 그룹장 양도

- **FR-022**: 그룹장은 현재 그룹 멤버에게 그룹장 권한 양도를 요청할 수 있어야 합니다.
- **FR-023**: 양도 대상자는 양도 요청을 수락하거나 거절할 수 있어야 합니다.
- **FR-024**: 양도 요청이 수락되면 시스템은 대상자를 그룹장으로, 이전 그룹장을 일반 멤버로 전환해야 합니다.
- **FR-025**: 양도 요청이 수락될 때까지 현재 그룹장은 그룹장 권한을 유지해야 합니다.
- **FR-026**: 그룹장은 대상자가 응답하기 전에 양도 요청을 취소할 수 있어야 합니다.
- **FR-027**: 시스템은 양도 대상자에게 즉시 알림을 전송해야 합니다.
- **FR-028**: 시스템은 동시에 한 명에게만 양도 요청을 보낼 수 있도록 제한해야 합니다.
- **FR-029**: 대기 중인 양도 요청이 있는 그룹장은 그룹을 떠날 수 없어야 합니다(먼저 요청 취소 필요).
- **FR-030**: 시스템은 30일 이상 응답이 없는 양도 요청을 자동으로 취소하고 그룹장에게 알림을 보내야 합니다.

#### 자동 승계 및 엣지 케이스 처리

- **FR-031**: 그룹장이 삭제되거나 강제로 그룹을 떠나면, 시스템은 다음 순서로 자동 승계해야 합니다:
  1. 가장 오래된 부그룹장
  2. 부그룹장이 없으면 가장 오래된 일반 멤버
  3. 멤버가 없으면 그룹을 자동 보관 처리
- **FR-032**: 보관된 그룹 복원 시 삭제된 멤버는 제외하고 활성 멤버만 복원해야 합니다.
- **FR-033**: 그룹장은 본인이 유일한 멤버인 경우에만 그룹을 떠날 수 있으며, 이 경우 그룹 삭제를 수행해야 합니다.
- **FR-034**: 부그룹장은 본인을 그룹에서 제거할 수 있으며, 이 경우 부그룹장 권한이 자동으로 회수됩니다.

#### 동시성 및 데이터 무결성

- **FR-035**: 시스템은 데이터베이스 수준의 고유 제약 조건으로 동시 그룹 생성 시 이름 중복을 방지해야 합니다.
- **FR-036**: 시스템은 동시 권한 변경 시 낙관적 잠금(optimistic locking) 메커니즘을 사용하여 데이터 무결성을 보장해야 합니다.
- **FR-037**: 권한 충돌 시 시스템은 사용자에게 "최신 정보로 새로고침 후 다시 시도하세요"라는 메시지를 표시해야 합니다.

#### 보안 및 감사

- **FR-038**: 시스템은 모든 권한 위반 시도를 로깅해야 합니다(누가, 언제, 어떤 작업을 시도했는지).
- **FR-039**: 시스템은 모든 그룹장 변경 이력을 감사 로그에 기록해야 합니다.
- **FR-040**: 시스템은 그룹 삭제 및 복원 이벤트를 감사 로그에 기록해야 합니다.

### 핵심 엔티티 (Key Entities)

- **그룹 (Group)**: 협업 공간을 나타내며, 고유한 이름(시스템 전체에서 고유, 대소문자 구분 없음), 설명, 생성 일시, 상태(활성/보관됨), 멤버 목록을 포함합니다. 하나의 그룹은 여러 멤버를 가질 수 있습니다.

- **그룹 멤버십 (Group Membership)**: 사용자와 그룹 간의 관계를 나타내며, 사용자 참조, 그룹 참조, 역할(그룹장/부그룹장/일반 멤버), 가입 일시를 포함합니다. 한 사용자는 여러 그룹의 멤버가 될 수 있습니다.

- **부그룹장 권한 (Vice-Leader Permission)**: 부그룹장에게 부여된 세부 권한을 나타내며, 멤버십 참조, 권한 유형들(멤버 관리 가능, 게시판 생성 가능, 콘텐츠 관리 가능), 권한 부여 일시를 포함합니다. 각 부그룹장은 개별적으로 다른 권한 조합을 가질 수 있습니다.

- **그룹장 양도 요청 (Leadership Transfer Request)**: 진행 중인 그룹장 양도 요청을 나타내며, 요청자(현재 그룹장) 참조, 대상자 참조, 그룹 참조, 상태(대기/수락됨/거절됨/취소됨), 요청 생성 일시, 응답 일시를 포함합니다. 하나의 그룹은 동시에 하나의 대기 중인 양도 요청만 가질 수 있습니다.

- **보관된 그룹 (Archived Group)**: 삭제된 그룹의 메타데이터를 나타내며, 그룹 참조, 보관 일시, 보관 수행자(그룹장 또는 시스템) 참조, 복원 일시(복원된 경우), 복원 수행자 참조(복원된 경우)를 포함합니다. 감사 및 복원 목적으로 사용됩니다.

- **감사 로그 (Audit Log)**: 중요한 그룹 이벤트를 추적하며, 이벤트 유형(그룹 생성/삭제/복원/그룹장 변경/권한 위반), 수행자 참조, 영향받는 그룹 참조, 영향받는 사용자 참조(있는 경우), 이벤트 일시, 세부 정보(JSON 형식)를 포함합니다.

## 성공 기준 *(필수)*

### 측정 가능한 결과 (Measurable Outcomes)

#### 사용자 경험 및 성능

- **SC-001**: 사용자는 그룹 생성 및 초기 설정(이름, 설명 입력)을 5분 이내에 완료할 수 있어야 합니다.
- **SC-002**: 그룹장은 부그룹장 지정 및 권한 설정을 2분 이내에 완료할 수 있어야 합니다.
- **SC-003**: 시스템은 권한 기반 접근 제어 검증을 95% 이상 1초 이내에 처리해야 합니다.
- **SC-004**: 그룹 멤버 목록 조회는 100명 이하의 멤버를 가진 그룹에서 2초 이내에 완료되어야 합니다.
- **SC-005**: 그룹장 양도 프로세스가 명확하고 직관적이어서 사용자가 추가 지원 없이 완료할 수 있어야 합니다(완료율 90% 이상).

#### 보안 및 무결성

- **SC-006**: 권한 위반 시도는 100% 차단되고 감사 로그에 기록되어야 합니다.
- **SC-007**: 그룹 이름 고유성이 100% 보장되어야 하며, 동시 생성 요청에서도 중복이 발생하지 않아야 합니다.
- **SC-008**: 모든 그룹장 변경 이벤트(양도, 자동 승계)는 100% 감사 로그에 기록되어야 합니다.
- **SC-009**: 보관된 그룹의 데이터 무결성이 100% 유지되어야 합니다(복원 시 손실 없음, 단 삭제된 사용자 제외).

#### 시스템 안정성

- **SC-010**: 시스템은 동시에 1,000명의 사용자가 그룹 관련 작업을 수행해도 성능 저하 없이 작동해야 합니다.
- **SC-011**: 동시 권한 변경 시도의 99% 이상이 데이터 무결성 손실 없이 처리되어야 합니다(충돌 시 적절한 오류 메시지 제공).
- **SC-012**: 자동 승계 메커니즘은 그룹장 부재 시 100% 작동하여 그룹 관리 연속성을 보장해야 합니다.

#### 비즈니스 가치

- **SC-013**: 사용자의 90% 이상이 첫 시도에서 그룹 생성 및 멤버 초대를 성공적으로 완료해야 합니다.
- **SC-014**: 권한 관리 관련 사용자 지원 요청이 월 10건 미만으로 유지되어야 합니다(명확한 오류 메시지 및 권한 표시).
- **SC-015**: 그룹 관리 기능의 사용자 만족도가 5점 만점에 4점 이상이어야 합니다.

## 의존성 (Dependencies)

이 기능은 다음 기능에 의존합니다:

- **001-user-auth** (사용자 인증 시스템): 사용자가 로그인되어 있어야 그룹 관련 기능을 사용할 수 있습니다. 사용자 ID 조회, 인증 상태 확인 등이 필요합니다.

## 가정 (Assumptions)

1. **알림 시스템**: 그룹장 양도 요청, 권한 변경 등의 알림을 전송할 수 있는 기본 알림 메커니즘이 있다고 가정합니다(이메일, 인앱 알림 등).
2. **사용자 삭제 처리**: 시스템에는 사용자 삭제(계정 탈퇴 또는 관리자 삭제) 메커니즘이 있으며, 사용자 삭제 시 적절한 이벤트가 발생하여 그룹 멤버십 정리가 가능하다고 가정합니다.
3. **동시성 제어**: 시스템 인프라가 낙관적 잠금 또는 비관적 잠금과 같은 동시성 제어 메커니즘을 지원한다고 가정합니다.
4. **데이터 보관 정책**: 보관된 그룹 데이터는 최소 1년간 보존되며, 이후 영구 삭제 정책은 별도의 데이터 보존 정책에 따른다고 가정합니다.
5. **게시판 시스템**: 부그룹장의 "게시판 생성" 권한은 향후 구현될 게시판 시스템에서 사용되며, 현재 단계에서는 권한만 정의한다고 가정합니다.
6. **관리자 권한**: 시스템 관리자는 보관된 그룹을 복원할 수 있는 별도의 권한 체계를 가진다고 가정합니다(일반 사용자와 구분).

## 범위 외 (Out of Scope)

이 기능에는 다음 사항이 포함되지 않습니다:

- **그룹 검색 및 공개 그룹 목록**: 모든 그룹은 초대를 통해서만 가입 가능하며, 공개 그룹 디렉토리나 검색 기능은 제공하지 않습니다.
- **그룹 카테고리 또는 태그 시스템**: 그룹을 분류하거나 태그를 지정하는 기능은 포함되지 않습니다.
- **그룹 통계 및 분석**: 그룹 활동, 멤버 참여도, 콘텐츠 통계 등의 분석 기능은 포함되지 않습니다.
- **그룹 템플릿**: 미리 정의된 그룹 구조나 설정을 사용하는 템플릿 기능은 포함되지 않습니다.
- **멤버 초대 승인 프로세스**: 초대된 사용자는 자동으로 그룹 멤버가 되며, 별도의 초대 승인 또는 가입 요청 프로세스는 없습니다.
- **그룹 가시성 설정**: 모든 그룹은 동일한 가시성(비공개)을 가지며, 공개/비공개 설정은 포함되지 않습니다.
- **대량 멤버 초대**: 한 번에 여러 사용자를 초대하는 기능은 포함되지 않으며, 개별 초대만 지원합니다.
- **역할 커스터마이징**: 그룹장, 부그룹장, 일반 멤버 외의 커스텀 역할을 생성하는 기능은 포함되지 않습니다.
